Lumex

Lumex is an implementation of the ray tracer described in Jamis Bruck’s awesome book The Ray Tracer Challenge.

 (require lumex) package: lumex

These predicates are provided instead of the default ones such as Vec2? and M3x3?.

Using the structures directly is unsafe in the sense that the flvs length are unchecked. This means that it is very much possible to create a Vec4 using a FlVector with too few elements.

This is just an example, please do not do this:

Example:
> (Vec4 [flvector 0.0])

(Vec4 #fl(0.0))

Note that the FlVector should have four components in this case.

If you have to, at least provide a FlVector of appropriate length:

Example:
> (Vec4 [flvector 0.0 0.1 0.2 0.3])

(Vec4 #fl(0.0 0.1 0.2 0.3))

It is usually not necessary to use the bare struct constructors but sometimes it is convenient to work with FlVectors directly so the possibility is there. Take utmost care when using them or strange things might happen. The recommendation is to use one of the constructor functions defined below.

These types are provided for reference. They are not intended to be used unless you are intimate with the library, writing lumex library code and/or know what you are doing.

These last two functions are targeted for client code and should be used when you want to find out whether a particar Vec4 value is a pos? or dir?.

The constructs above are safe so that it is impossible to contstruct invalid values. That is, where the internal FlVector is too short to support the dimensions of the type.

Use these constructors if possible unless you are doing very low level stuff and know what you are doing.

Examples:
> (vec2 2.0 3.0)

> (m3x3 0.0 0.1 0.2

        2.0 2.1 2.2)

(M3x3 #fl(0.0 0.1 0.2 1.0 1.1 1.2 2.0 2.1 2.2))

Colors are just a variant of Vec3 values but it is useful to classify them and to be able to test for them. When doing operations on them, we will mostly use basic vec3 functions.

procedure

(color r g b) → Color

  r : Real
  g : Real
  b : Real
Constructor function for Color values.

Examples:
> (color 0.2 0.5 0.123)

(Color #fl(0.2 0.5 0.123))

> (color 2 3 1)

(Color #fl(2.0 3.0 1.0))

Positions and directions are both Vec4 values but they are conceptually different and it helps to keep them separate when doing math on them as well. For that reason, most client code should be using the pos and dir constructors instead of the more low-level vec4 constructor.

procedure

(pos x y z) → Vec4

  x : Real
  y : Real
  z : Real
Constructs a position vector. Position vectors have their w component set to 1.0.

procedure

(dir x y z) → Vec4

  x : Real
  y : Real
  z : REal
Constructs a direction vector. Direction vectors have their w component set to 0.0.

Examples:
> (define-values (p v)
    (values (pos 1 2 3) (dir 1 2 3)))
> p

(Vec4 #fl(1.0 2.0 3.0 1.0))

> v

(Vec4 #fl(1.0 2.0 3.0 0.0))

> (vec4-w p)

1.0

> (vec4-w v)

0.0

> (pos? p)

#t

> (pos? v)

#f

> (dir? p)

#f

> (dir? v)

#t

When doing arithmetic, lumex usually tries do do the sensible thing with regards to linear algebra. This means for example that in some cases, doing operations with two pos? values might result in a dir? value or vice versa.

The above indexers are safe in the sense that the type system will prevent calling them with a row and col combination that would be out of bounds.

floating point division

March 14, 2018

Suppose we have two floats a and b with a = 1 * 2**0 = 1 and b = 3 * 2**0 = 3 and we want to divide these. Remember that the mantissa values are stored as int so we have to sort this out using integer division.

Let’s start out by writing that a / b = (1 * 2**0) / (3 * 2**0) = 1/3 * 2**0. Clearly we can’t store 1/3 as our new mantissa since this is a rational and not an integer number. We need two numbers x and y so that we can write a / b * 2**0 = x * 2**y where x and y are both integer numbers. One thing we can observe straight away is that the only way to accomplish this is to use big values of x with small values of y if we hope to come close to a decent approximation of 1 / 3 * 2**0.

Let’s try out something:

1/3 =  (0 + 1 / 3) * 2**+0
    =  (1 + 1 / 3) * 2**-2
    =  (5 + 1 / 3) * 2**-4
    = (21 + 1 / 3) * 2**-6
    = (85 + 1 / 3) * 2**-8
    = ...

Now there’s clearly a pattern here and also, once the numbers get bigger, the 1/3 part starts to matter less and less. In fact, we have to drop it at some point. Where that point is depends on the precision we want. Let’s drop the 1 / 3 part completely and just focus on the whole part of the mantissa and the exponents. We’ll call them x and y respectively.

Now we create a function m so that m(i) = (x, y). That is, for some iteration value of i we get a tuple of x (the mantissa) and y the exponent (where i >= 0). For example:

m(0) = ( 0, +0)
m(1) = ( 1, -2)
m(2) = ( 5, -4)
m(3) = (21, -6)
m(4) = (85, -8)
...

Here’s a possible definition for the x (mantissa) part of the function:

int mx(int i) => i <= 0 ? 0 : mx(i - 1) + (1 << ((i - 1) * 2));

Note that if n <= 0 we just return 0 this not only makes some sense but also it’s a nice terminating condition for our recursive definition.

The y part is a lot easier:

int my(i) => -(i * 2);

And with these we can also easily compose our function m:

(int, int) m(i) => (mx(i), my(i));

Let’s see what happens when we plug those values in our original function expression. First we define a function f that calculates our expression x * 2**y:

double f(int x, int y) => x * Math.Pow(2, y);

Then we plug in the values:

i               x       y       f(x, y)
---------------------------------------------
0               0       0       0
1               1      -2       0.25
2               5      -4       0.3125
3              21      -6       0.328125
4              85      -8       0.33203125
...
15      357913941     -30       0.333333333022892

The approximation of 1 / 3 gets better and better! Note that in this case we run into overflow problems around i > 16 but that’s mostly just because we’re using int in this case. Since mx(16) = 1431655766 that’s already pretty close to int.MaxValue and multiplying it by 2 would definitely bring us over.

If we would use BigInteger values then we could in theory support floating point values of arbitrary (large) precision.
